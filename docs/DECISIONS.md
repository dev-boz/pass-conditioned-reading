# DECISIONS.md — every parameter invented, with its derivation

The parent proposal and v0.3 outline define the schedule *shape* but almost no
concrete values (see `docs/INVENTORY.md`, "code vs prose" table). Everything
below was chosen here. Nothing in this file is pre-existing schedule design.

| # | Parameter | Value | Derivation / spec line it leans on |
|---|---|---|---|
| D1 | K (passes per doc, E0) | **8** | Brief: "prefer documents long enough that K ≥ 6 is meaningful." Parent README §4 "Cost honesty": ~30 passes for 80–150K-token sources ⇒ roughly one pass per 3–5K source tokens; GovReport docs here are 8–24K tokens ⇒ K=8. Fixed across docs so E0's exact-k target is a clean 8-class problem. |
| D2 | V (per-view token budget) | **512** | Parent §4 window invariant `|s|+|v|+overhead ≤ W`: V constant by design. 512 gives pass-1 compression ≥16× on the 8K-token floor (D3) and verbatim final slices big enough to contain whole passages. |
| D3 | Corpus filter | GovReport, 8 192–24 000 tokens, n=300 | Brief offers GovReport/BookSum/QASPER/arXiv and 200–500 docs. GovReport chosen: long single-narrative reports, public domain, clean text, the standard long-doc summarization corpus (justified in E0 RESULTS.md). Floor = 16×V so heavy early compression is real; cap bounds CPU cost. |
| D4 | Compression ladder | slice_len_k = N·(V/N)^((k−1)/(K−1)); ratio_k = slice_len_k/V, geometric from N/V down to 1 | Parent §4 sketch: `sigma_1 = max, decreasing to sigma_K = 0`, slice "broad early, narrow late," coupled (§3 step 1). Geometric interpolation between the two prose-specified endpoints; the interpolation *shape* is invented. |
| D5 | Slice placement (k ≥ 2) | deterministic hash-RNG per (doc_id, k), uniform over valid starts | Parent prose never says *where* narrow slices go. A left-to-right sweep would correlate source position with pass index and hand E0 a shortcut that is a schedule-design artifact, not a compression-level signal — biasing E0 toward a spurious "time-agnostic" call. Placement is therefore position-decorrelated but fully deterministic (reproducible). The production schedule may eventually sweep systematically; if so, E0 must be re-run on that schedule. Conservative direction: this choice can only make position *harder* to predict. |
| D6 | Stand-in compressor | extractive SumBasic-style (idf-damped salience + mild lead prior, greedy to budget, original order) | Production C is teacher-LLM summarization (parent §5 "Data generation"); no zero-cost inference path exists for 2 400 LLM calls (inventory). Brief contingency: run the stand-in, label `E0-provisional`. Deterministic by construction. |
| D7 | `compression_ratio` excluded from Tier-A features | excluded | The brief lists "compression ratio" as a Tier-A hand feature, but the stored ratio = slice_tokens/view_tokens is **schedule metadata, not view content** — it is a near-deterministic function of k under D4, so feeding it to the classifier makes E0 circular (the label leaks through the feature, not through the view). E0's question is "predict (k,K) from `v_k` alone." Tier A therefore uses only view-text-computable features; view length in tokens (which is the text-side shadow of the ratio) stays in. Deviation flagged in RESULTS.md. |
| D8 | Bucket definition | t=(k−1)/(K−1); early t<1/3 → k∈{1,2,3}; mid t<2/3 → k∈{4,5}; late → k∈{6,7,8} | Brief specifies {early, mid, late} but not the boundary; terciles of normalized position. Class imbalance (3/2/3) reported alongside the 1/3 chance baseline. |
| D9 | Split | 70/15/15 by doc_id, shuffled with seed 42 | Brief suggests exactly this. |
| D10 | E0 gate thresholds | Tier-B bucket acc ≥90% → time-agnostic; ≤70% → signal-carrying; else mixed | Verbatim from the brief; pre-registered in `experiments/e0_view_classifier/config.yaml` before the run. |
| D11 | P1/P2 model + inference path | Qwen2.5-7B-Instruct, Q4_K_M GGUF, llama.cpp CPU server | Brief: "7–8B class; use whatever inference path the inventory finds." Inventory found none ⇒ local llama.cpp, $0 spend. Qwen matches the parent's experimental lineage (parent §6: Qwen 2.5 generalists). |
| D12 | P1 schedule | K=6, V=512, same D4 ladder; doc = one corpus doc ~9–10K tokens | K=6 keeps CPU wall-clock sane (18 generations) while ≥6 per brief. |
| D13 | Output-detail ladder (P1 coupled/output arms) | passes 1–2 scaffold; 3–4 concrete-abstractive; 5–6 verbatim integration | v0.3 §3 Variant 2 names the three stages ("scaffold, then concrete-abstractive, then verbatim integration") but not their boundaries; even thirds of the schedule. |
| D14 | Scaffold budget B_s | **600 tokens** | v0.3 §3: "scaffold length is capped at an explicit budget B_s (a schedule parameter)" — no value given anywhere. 600 ≈ V+overhead; bounded-pass-1 cost O(B_s) preserved. |
| D15 | Edit-op concrete syntax | line-oriented: `ADD <id|END>: text` / `REPLACE <id>: text` / `REMOVE <id>` / `NO_CHANGE`; state = numbered entries S1, S2, … | Parent §3 step 3 names the four ops, no syntax anywhere. Line-oriented chosen for parseability by 7B models (the V1 failure mode was malformed ops). Parser in `src/pca/editops.py`; validity = parseable lines with resolvable ids / total op lines. |
| D16 | State budget (P1) | 1 800 tokens; runner warns (does not truncate) when exceeded | Parent §4: state bounded by edit mix, "expected behavior shifts … as the state approaches budget." With W=12 288 (D17): state + V + prompt overhead comfortably ≤ W. |
| D17 | llama.cpp context window W | 12 288 | Must hold full P1 doc (~9–10K tokens) for the output-staged-only arm's full-resolution view. RAM-feasible at Q4 on 16 GB. |
| D18 | Edit budget per pass (P1/P2) | max 12 ops, ≤700 generated tokens | Parent §3/v0.3: "output cost per pass is bounded by the edit budget" — value unspecified. Sized to CPU generation speed. |
| D19 | P2 eval items | the K=6 coupled-arm items from P1's document (s_{k-1} from the reference coupled run), each run with and without the position field; neutral prompt otherwise | Brief contingency: "If the V1 PoC eval set isn't locatable in Phase 0, build the harness against the P1 document and note it." It is not locatable (inventory). |
| D20 | Position-appropriateness heuristic (P2) | early (t<1/3): ADD-fraction ≥ 0.5 counts as scaffold-like; late (t≥2/3): REPLACE+REMOVE fraction ≥ 0.3 counts as edit-like; mid passes excluded from the headline | Brief: "a simple position-appropriateness check (e.g., scaffold-like vs edit-like output at early vs late passes)." Thresholds invented; reported as a heuristic, never as a claim. |
| D21 | Tokenizer convention | tiktoken cl100k_base everywhere | Any consistent convention works; chosen for speed and determinism on CPU. |
| D22 | Parser v2.1: named-ADD resolution with alias map (2026-06-13; v2.0 same-day) | (1) ops may reference ids created by earlier ADDs in the same response; (2) an ADD whose anchor does **not** resolve is a *named append* — models name the entry they are creating, sequentially from arbitrary origins (S1, the system prompt's example `S3`, max-visible+1 — never the harness's hidden counter) — and later ops in the response resolve that name via an alias map to the actually-assigned id; (3) REPLACE/REMOVE must still resolve (state ∪ same-response ∪ alias). Consequence: ADD validity is grammar-only; REPLACE/REMOVE carry the reference-validity signal. | The first P1/P2 runs showed the dominant 7B malformation was models treating `ADD <id>` as naming, not as insert-after (parser v1 rejected ~all of coupled pass 1). the maintainer directed the fix, offering same-response resolution or anchorless ADD; v2.0 implemented strict same-origin self-naming and **under-fixed** (P3 k=1 0/14 with naming origin copied from the prompt example; P1 coupled pass 6 5/14 with origin at max-visible+1), so v2.1 combines both options: resolvable anchors keep insert-after, unresolvable ADD anchors become named appends (the anchorless-ADD semantics). v1 numbers archived under `*_parser-v1/`, v2.0 under `*_parser-v2.0/`; **every reported P1/P2/P3 number is parser-v2.1.** |
| D23 | P3 rubric stage mapping & confidence line | rubrics per D13 stage boundaries (passes 1–2 scaffold / 3–4 integrate-REPLACE / 5–6 verbatim REPLACE-REMOVE, NO_CHANGE allowed); arm (c) requires final line `confidence: 0.0–1.0` (state adequacy for the task, parent §3's confidence signal); calibration = per-item confidence vs validity, Pearson r over n=6 | the maintainer's P3 spec gives the rubric content per stage; exact 2–3-line wording and the confidence-line format are invented here (`experiments/p3_prompt_rubric/config.yaml`). |
| D24 | Mean formulas | every reported mean states its formula; validity means reported both **micro** (Σ valid / Σ candidate lines, op-weighted) and **macro** (unweighted mean of per-pass rates) | First OBSERVATIONS.md quoted means that recomputed from neither formula (the maintainer caught it); all tables now machine-generated from the run logs/JSON. |
| D25 | P4 schedule: two-phase coverage-complete coupled (2026-06-13) | Phase A — R coarse global-scaffold chunks (each ~N/R turns → V, ~5×); Phase B — verbatim sweep of ceil(N/V) contiguous turn-tiles (≤V, σ=0) covering all 390 turns. Turn-granular (the manifest is turn-indexed), so coverage-timing is a turn-range intersection. **Not** E0's hash placement (D5) — here coverage is the object of study. | Parent README §4 cost-honesty: coarse→fine, "every region visited at σ=0 by the end," pass count ~|x|/verbatim-slice. Output-staged-only arm **dropped** (advisor): "full-resolution view every pass" is ill-defined at 80K (the source doesn't fit any window) — incoherent, not merely expensive. |
| D26 | P4 K, V, R | V=3000, R=5 → **K=32** (6 ramp + 26 verbatim) for the 72.9K-token (tiktoken) source | "K calibrated to length" per parent's ~30-pass/80–150K calibration. V=3000 chosen for a real Phase-A compression contrast (~5×) while K stays near 30; verified on paper (`dryrun.py`): 390/390 turns covered, F7@turn153 first verbatim at pass 17, F4 recoverable (pass 1) before F8 (pass 4). |
| D27 | P4 output-demand ladder | scaffold / concrete / verbatim by pass-fraction boundaries [0.34, 0.67] (rubric wording = P3's, the E1 prompt floor) | P3's 1–2/3–4/5–6 thirds (D23) don't transfer to K=32; generalized to pass fraction. |
| D28 | P4 budgets & scoring | state budget 4000 tok (warn, not truncate), ctx 16384; dense baseline ctx 32768, 30K head+tail fill (50/50, middle omitted with a marker — the most generous single-pass read). **Final-state recall = normalized component matcher** (`facts_scoring` in config); **coverage-timing = manifest exact strings** on the source. | Advisor: exact-substring on the paraphrasing state would repeat the parser exact-match trap (the state says "his grandmother", "8000", "299"). Locked here because the same compressor must generate all training data — switching it later invalidates cross-experiment comparability. | User directive (2026-06-13): "lock this choice in DECISIONS.md — the same compressor must generate all future training data." Model id/pricing confirmed via the `claude-api` skill, not memory. **E0-final cost estimate** (from the E0 per-pass median slice tokens): 300 docs × 8 passes ≈ 9.9M input + 1.2M output tokens ⇒ **≈ US$16 real-time / ≈ US$8 batched**. On the subscription/OAuth path the same work draws pro-plan usage (not directly dollar-metered); 2 400 sequential Haiku calls is a large batch, so **confirm before running**. **Status (2026-06-14): RESOLVED — invocation path = `claude -p --model haiku`.** Tried Haiku **subagents** first (the maintainer's idea — `Agent(model="haiku")` under session OAuth, no key): proved the concept but was operationally unreliable at scale (only 4/30 ten-doc batches clean; autonomous workers improvised — looked for an API key, wrote 1-word stubs, wrote 2,100-word essays, and one **misaligned view_text to the wrong slice** (rb_34), a data-validity risk a length/distinctness audit can't catch; the run also exhausted the 5h quota). Cost is plan usage (Haiku-tier) over 2,400 calls; resumable across quota resets. |
